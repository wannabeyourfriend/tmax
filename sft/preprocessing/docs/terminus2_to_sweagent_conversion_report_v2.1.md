## 1. Executive Summary

We have approximately 381,000 agentic traces in Terminus-2 format from two source datasets that we need to convert into SWE-agent style format for supervised fine-tuning:

1. **`open-thoughts/OpenThoughts-Agent-v1-SFT`** — ~15,200 traces (nl2bash + InferredBugs tasks, generated by GLM-4.6)
2. **`nvidia/Nemotron-Terminal-Corpus`** — ~366,000 traces across 4 subsets (Math/Code/SWE dataset adaptations + skill-based synthetic tasks, generated by DeepSeek-V3.2)

Both datasets use the same Terminus-2 harness and share an identical conversation-level schema. The goal is to train a model that, at inference time, produces structured `tool_calls` with `function.name` / `function.arguments` and consumes `role: "tool"` observation messages, while also using `<think>` blocks for chain-of-thought reasoning.

The core challenge: these two formats encode the same underlying behavior (think, act, observe, repeat) but in completely different surface token distributions. Terminus-2 embeds everything in flat `user`/`assistant` string content with JSON-structured actions, while SWE-agent uses the OpenAI function-calling convention with separate message roles and structured tool invocations.

This document specifies every detail of the conversion so that the implementation can proceed without ambiguity.

### 1.1 Key Design Decision: Persistent Shell Harness

**Our inference harness will use a persistent shell session**, not the stateless `subprocess.run` model used by mini-swe-agent. This is a deliberate architectural decision driven by the nature of our training data.

**The problem with stateless execution:** mini-swe-agent executes every action as an independent subprocess — working directory, environment variables, and shell state reset between calls. The Terminus-2 traces were generated against a persistent tmux session where state accumulates across turns. Commands in turn N+1 implicitly depend on `cd`, `export`, `source`, and other state-modifying commands from turn N. Converting these traces for a stateless harness would require simulating a bash interpreter to track accumulated state and prepend `cd /accumulated/path && export VAR=val &&` to every command — a fragile, error-prone, and fundamentally unsolvable problem in the general case.

**The persistent shell solution:** By adopting a persistent shell at inference time, the Terminus-2 traces are semantically correct as-is. A `cd /testbed` in turn 1 genuinely persists to turn 2. Environment variables set via `export` survive across calls. The terminal output (including prompts showing the current working directory) is accurate rather than misleading. Special keystrokes like `C-c` (Ctrl+C) have a natural home as legitimate operations in an ongoing session.

**Impact on conversion:** This decision dramatically simplifies the conversion pipeline. There is no need to track shell state, no need to drop or rewrite commands, and no semantic correctness gap between the training data and inference environment. The conversion becomes primarily a structural reformatting task (JSON-in-content → structured tool_calls) rather than a semantic transformation.

**Trade-off:** The inference harness is slightly more complex than `subprocess.run` (requires managing a tmux or pexpect session), but this is well-understood engineering. See Appendix C for implementation guidance.

## 2. Source Formats: Terminus-2 Traces

Both source datasets use the Terminus-2 harness and share an identical conversation-level schema. They differ in dataset-level metadata columns, teacher models, task domains, and scale. This section documents the shared schema first, then per-dataset specifics.

### 2.1 Shared Conversation-Level Schema

Each element in a trace's `conversations` list is a dict with exactly two keys:

```json
{
  "content": "string",
  "role": "user" | "assistant"
}
```

There is **no** `role: "system"` message. There is **no** `role: "tool"` message. There are **no** `tool_calls`, `tool_call_ids`, `thought`, `action`, `agent`, or `message_type` fields. Only `role` and `content`. This is true for both datasets.

### 2.2 Dataset A: `open-thoughts/OpenThoughts-Agent-v1-SFT`

**Scale:** ~15,200 traces, 2-64 messages each, up to 32 turns.

**Dataset-level columns:**

| Column | Type | Description |
|--------|------|-------------|
| `conversations` | `list[dict]` | The conversation trace (2-64 messages) |
| `agent` | `str` | Always `"terminus-2"` |
| `model` | `str` | Always `"QuantTrio/GLM-4.6-AWQ"` |
| `model_provider` | `str` | Always `"hosted_vllm"` |
| `date` | `str` | ISO timestamp of trace generation |
| `task` | `str` | Task identifier (e.g., `"task_0"`, `"task_1000"`) |
| `episode` | `str` | Episode identifier (e.g., `"episode-1"`) |
| `run_id` | `str` | UUID of the generation run |
| `trial_name` | `str` | Combined task+random suffix (e.g., `"task_0__X49A9se"`) |

**Teacher model:** `QuantTrio/GLM-4.6-AWQ` — a quantized version of GLM-4.6.

**Task domains:** nl2bash (shell command tasks) and InferredBugs (C#/Java bug-fixing tasks).

**Maximum context length at generation:** 64K tokens.

### 2.3 Dataset B: `nvidia/Nemotron-Terminal-Corpus`

**Scale:** ~366,000 traces total across 4 subsets, 2-38 messages each.

**Subsets:**

| Subset | Rows | Description |
|--------|------|-------------|
| `dataset_adapters` | ~226,000 | Math, Code, and SWE datasets transformed into terminal-based format |
| `skill_based_easy` | ~44,800 | Synthetic tasks from a structured taxonomy of terminal skills (easy difficulty) |
| `skill_based_medium` | ~89,300 | Synthetic tasks (medium difficulty) |
| `skill_based_mixed` | ~5,690 | Synthetic tasks (mixed difficulty) |

**Dataset-level columns (all subsets share this schema):**

| Column | Type | Description |
|--------|------|-------------|
| `conversations` | `list[dict]` | The conversation trace (2-38 messages) |
| `agent` | `str` | Always `"terminus-2"` |
| `model` | `str` | Always `"deepseek-ai/DeepSeek-V3.2"` |
| `model_provider` | `str` | Always `"hosted_vllm"` |
| `date` | `str` | ISO timestamp of trace generation |
| `task` | `str` | Task identifier (e.g., `"data_processing_task_0717"`) |
| `episode` | `str` | Episode identifier (e.g., `"episode-4"`) |
| `run_id` | `str` | UUID of the generation run |
| `trial_name` | `str` | Combined task+random suffix (e.g., `"data_processing_task_0717__qPycjT6"`) |
| `enable_thinking` | `bool` | **Nemotron-only field.** Whether thinking was enabled during trace generation. Observed as always `true` in `skill_based_mixed`. |

**Teacher model:** `deepseek-ai/DeepSeek-V3.2` — a DeepSeek thinking model.

**Task domains:** Data querying, model training, data processing, debugging, software engineering, and adapted Math/Code/SWE tasks.

**Key differences from OpenThoughts:**
1. **The `enable_thinking` column** is present in Nemotron but absent in OpenThoughts. This field records whether thinking mode was enabled during generation. For conversion purposes, we do not use this field to control output — all converted traces get `<think>` blocks regardless, since we are training a thinking model. However, we preserve it in output metadata for provenance.
2. **Much larger scale.** The `dataset_adapters` subset alone is 226K rows. The Parquet files for this subset are very large (individual row groups exceed 300MB), which may cause issues with the HuggingFace dataset viewer and require chunked loading (see Section 8).
3. **Different teacher model.** DeepSeek-V3.2 may produce different JSON formatting patterns than GLM-4.6 — potentially cleaner JSON (fewer extraction failures) or different prose style in analysis/plan fields. The JSON extraction cascade in Section 5.4 must handle both.
4. **Different task naming patterns.** OpenThoughts uses `task_0`, `task_1000`, etc. Nemotron uses domain-prefixed names like `data_processing_task_0717`. The `trial_name` field (used for deterministic ID generation) is unique per-trace in both datasets, so no collision risk.
5. **Shorter max conversation length.** Nemotron traces go up to 38 messages (19 turns) vs OpenThoughts' 64 messages (32 turns).

### 2.4 Shared Conversation Structure

Both datasets follow the same alternating pattern:

```
Message 0: role=user     System prompt + task description + initial terminal state
Message 1: role=assistant  JSON blob with analysis/plan/commands
Message 2: role=user     Terminal output from executed commands
Message 3: role=assistant  JSON blob with analysis/plan/commands
Message 4: role=user     Terminal output (possibly with "Are you sure?" confirmation)
Message 5: role=assistant  JSON blob (final, with task_complete: true)
```

### 2.5 Message 0: The Combined System+Task+State Message

The first user message concatenates three logical sections into a single string, without explicit delimiters:

```
[SECTION A: System Prompt - instructions for JSON output format]
...Format your response as JSON with the following structure...
...Required fields: "analysis", "plan", "commands"...

[SECTION B: Task Description]
Task Description:
# Terminal Automation Request
**TERMINAL ENVIRONMENT**: Ubuntu 24.04 with bash...
## Goal
Search for regular files...
## Requirements
- Run the command block...

[SECTION C: Initial Terminal State]
Current terminal state:
Current Terminal Screen:
root@e7aef04b:/workspace#
[blank lines filling 40-line terminal buffer]
```

The boundary between Section A and Section B is the literal string `"Task Description:\n"`. The boundary between Section B and Section C is the literal string `"Current terminal state:\n"`.

**Important:** Both datasets use the same Terminus-2 system prompt template. However, the task description content differs significantly — OpenThoughts tasks tend to be simpler shell operations, while Nemotron `dataset_adapters` tasks may involve multi-step programming or math problems adapted for terminal execution. The conversion pipeline handles both identically; the task content is preserved verbatim.

### 2.6 Assistant Messages: The JSON Action Blob

Each assistant message contains a JSON object, sometimes preceded or followed by free-form prose. The JSON object has this schema:

```json
{
  "analysis": "string - reasoning about current state",
  "plan": "string - planned next steps",
  "commands": [
    {
      "keystrokes": "string - literal keystrokes to send to terminal",
      "duration": 0.1
    }
  ],
  "task_complete": true | false
}
```

**Critical details about the JSON blob:**

- `analysis` and `plan` are always present and are free-form strings.
- `commands` is an array of 1 or more command objects.
- `keystrokes` contains the exact string to send, including `\n` for Enter. Special sequences include `C-c` (Ctrl+C), `C-d` (Ctrl+D), and empty strings with nonzero `duration` (waiting).
- `duration` is a float in seconds. Values observed: `0.1` (fast commands like `ls`, `cd`, `echo`), `1.0` (moderate commands like `find`, `gcc`), up to `60.0` (long-running processes).
- `task_complete` is optional; when absent, defaults to `false`. When `true`, signals the agent considers the task finished.

**Known variations in assistant content:**

1. **Clean JSON only:** The content is a valid JSON object and nothing else.
2. **Prose before JSON:** The model outputs natural language reasoning before the JSON blob (e.g., "Let me analyze what has happened:\n\n1. I can see..." followed by `{...}`).
3. **Prose after JSON:** Less common, but some traces have trailing text after the closing `}`.
4. **Leading/trailing whitespace or newlines around the JSON.**
5. **Malformed JSON:** Missing closing braces, unescaped quotes within command strings, trailing commas. These are generation failures from the teacher model.

**Per-teacher-model expectations:**
- **GLM-4.6 (OpenThoughts):** Known to occasionally produce prose before JSON. JSON quality is generally good but not perfect.
- **DeepSeek-V3.2 (Nemotron):** As a thinking model, may produce longer analysis/plan text. JSON formatting quality should be validated empirically by running the extraction cascade on a sample of each subset before full-scale processing.

### 2.7 User Messages (Turn > 0): Terminal Output

Subsequent user messages contain the terminal output from the executed commands. They follow one of two patterns:

**Pattern A — Standard output:**
```
Current terminal state:
New Terminal Output:
root@hostname:/path# <command that was run>
<output lines>
root@hostname:/path#
[blank lines to fill 40-line terminal buffer]
```

**Pattern B — Confirmation prompt (final turn only):**
```
Current terminal state:
New Terminal Output:
<terminal output from commands>
[blank lines]
Are you sure you want to mark the task as complete? This will trigger your solution to be graded and you won't be able to make any further corrections. If so, include "task_complete": true in your JSON response again.
```

The confirmation prompt appears when the previous assistant message had `task_complete: true`. The harness is asking for re-confirmation.

## 3. Target Format: SWE-Agent Style with Persistent Shell

### 3.1 Conversation-Level Schema

Each message is a dict with a variable set of keys depending on role:

**System message:**
```json
{
  "role": "system",
  "content": "string"
}
```

**User message:**
```json
{
  "role": "user",
  "content": "string"
}
```

**Assistant message (with tool use):**
```json
{
  "role": "assistant",
  "content": "string - prose/reasoning visible to user",
  "thought": "string - chain-of-thought reasoning (will become <think> block)",
  "tool_calls": [
    {
      "function": {
        "name": "string - tool name",
        "arguments": { "key": "value" }
      },
      "id": "string - unique identifier",
      "type": "function"
    }
  ]
}
```

**Assistant message (no tool use, e.g., final summary):**
```json
{
  "role": "assistant",
  "content": "string - final response text",
  "thought": "string - optional reasoning"
}
```

**Tool result message:**
```json
{
  "role": "tool",
  "content": [
    {
      "type": "text",
      "text": "string - tool execution output"
    }
  ],
  "tool_call_ids": ["string - matching the tool_call id"]
}
```

### 3.2 The `thought` Field and `<think>` Blocks

Since we are training a thinking model, the `thought` field on assistant messages will be rendered as `<think>...</think>` blocks by the chat template. During training, the Qwen3.5 chat template handles this as follows:

- If `message.reasoning_content` is a string, it uses that.
- Otherwise, if the `content` contains `<think>...</think>`, it parses it out.
- The template preserves `<think>` blocks only for assistant messages after the last real user query (the "rolling checkpoint" behavior).

**Important:** The Qwen3.5 chat template uses the field name `reasoning_content`, not `thought`. The conversion pipeline must use the correct field name for whichever training framework is being used. If using Llama-Factory or a custom training script, verify which field name the data loader expects and map accordingly. This document uses `thought` as a logical name; the actual serialization key should be confirmed with the training pipeline owner.

### 3.3 Tool Schema

The system prompt in the target format must define the available tools. For converted Terminus-2 data, we define two tools:

**Tool 1: `bash`**
```json
{
  "type": "function",
  "function": {
    "name": "bash",
    "description": "Execute one or more commands in the persistent terminal session. The terminal state (working directory, environment variables, background processes, shell history) persists between calls. For multiple commands, separate with newlines. Special keystrokes like C-c (Ctrl+C) and C-d (Ctrl+D) are supported.",
    "parameters": {
      "type": "object",
      "properties": {
        "command": {
          "type": "string",
          "description": "The command(s) to execute. For multiple commands, separate with newlines."
        }
      },
      "required": ["command"]
    }
  }
}
```

**Tool 2: `submit`**
```json
{
  "type": "function",
  "function": {
    "name": "submit",
    "description": "Submit the solution and mark the task as complete.",
    "parameters": {
      "type": "object",
      "properties": {},
      "required": []
    }
  }
}
```

Note that the `bash` tool description explicitly documents persistent state. This is important — it teaches the model that `cd` in one call affects the next, which is exactly the behavior present in the Terminus-2 training data. At inference time, the persistent shell harness fulfills this contract.

### 3.4 The `arguments` Field: Dict, Not String

**This is critical for Qwen3.5 chat template compatibility.**

The `tool_calls[].function.arguments` field MUST be a Python dict (JSON object), NOT a JSON-encoded string. The Qwen3.5 chat template iterates over arguments using `tool_call.arguments|items`, which requires a mapping type. A JSON string like `"{\"command\": \"ls\"}"` will cause the template to fail.

Correct:
```json
"arguments": {"command": "ls -la"}
```

Incorrect:
```json
"arguments": "{\"command\": \"ls -la\"}"
```

If the training framework serializes conversations to JSON files, the arguments should be nested objects, not escaped strings. Verify this after conversion by running a sample through the actual chat template and confirming valid output.

### 3.5 The `tool_call_ids` Field

Each tool result message must have a `tool_call_ids` list containing exactly one string that matches the `id` field of the corresponding `tool_calls` entry. IDs should be unique within a conversation. We generate them deterministically (see Section 5.6).

## 4. System Prompt Replacement Strategy

### 4.1 Rationale

The Terminus-2 system prompt instructs the model to output JSON with `analysis`, `plan`, `commands`, and `task_complete` fields. We are converting away from this output format, so including the original system prompt would create contradictory training signal — the prompt says "output JSON" but the target tokens show tool_calls. We must replace it entirely.

### 4.2 Replacement System Prompt Template

The replacement system prompt should:

1. Describe the agent's role and environment.
2. Define the available tools (`bash`, `submit`).
3. Describe the persistent shell behavior (state persists between calls).
4. Instruct the model to reason in `<think>` blocks before acting.
5. Not reference JSON output format, `keystrokes`, `duration`, or any Terminus-2 concepts.

**Provide the actual system prompt text as a separate configuration file** (`system_prompt.txt`) that the conversion pipeline reads at startup. This allows iteration on the prompt without changing code. A reference template is:

```
You are an AI assistant tasked with solving command-line tasks in a Linux environment.

You have access to the following tools:

1. `bash` - Execute commands in a persistent terminal session. Working directory, environment variables, and shell state persist between calls.
2. `submit` - Submit your solution when the task is complete.

Think step-by-step about the current state and what needs to be done before taking action. When you are confident the task is complete, use the submit tool.
```

**Note:** The exact system prompt wording should be reviewed and aligned with whatever system prompt your inference harness uses, to minimize distribution shift between training and inference. The key addition vs v1.0 is the explicit mention of persistent state, which gives the model permission to rely on accumulated shell state across turns.

## 5. Conversion Logic: Step by Step

### 5.1 High-Level Pipeline

```
For each row in the dataset:
  1. Extract conversations list
  2. Parse Message 0 (split system prompt / task / terminal state)
  3. Replace system prompt
  4. Build initial user message from task description + terminal state
  5. For each subsequent (assistant, user) pair:
     a. Parse assistant JSON (robust extraction)
     b. Build thought from analysis + plan
     c. Build tool_calls from commands
     d. Build tool result from next user message
     e. Handle task_complete → submit
  6. Handle edge cases (confirmation turns, malformed JSON, etc.)
  7. Emit converted conversation
  8. Record conversion metadata (success/failure, warnings)
```

### 5.2 Step 2: Parsing Message 0

The first user message must be split into three sections. Use the following extraction logic:

```python
content = messages[0]["content"]

# Split on "Task Description:\n" to separate system prompt from the rest
TASK_DELIM = "Task Description:\n"
if TASK_DELIM in content:
    system_prompt_section = content[:content.index(TASK_DELIM)]
    remainder = content[content.index(TASK_DELIM):]
else:
    # Fallback: no task delimiter found
    # Flag this trace for manual review
    system_prompt_section = ""
    remainder = content

# Split remainder on "Current terminal state:\n"
STATE_DELIM = "Current terminal state:\n"
if STATE_DELIM in remainder:
    task_description = remainder[:remainder.index(STATE_DELIM)]
    terminal_state = remainder[remainder.index(STATE_DELIM):]
else:
    task_description = remainder
    terminal_state = ""
```

**Edge cases to handle:**
- The delimiter strings may have minor variations (extra whitespace, different casing). Use `.strip()` on sections after splitting.
- Some traces may not contain `"Task Description:\n"` at all (e.g., InferredBugs tasks may format differently). Flag these and inspect a sample before deciding on a fallback strategy.
- The terminal state section contains 40 lines of terminal buffer, most of which are blank. Preserve it as-is (the model needs to learn to parse terminal state, and the prompt line showing the working directory is accurate contextual information in our persistent shell model).

### 5.3 Step 3-4: Emit System and User Messages

```python
converted = []

# System message (replacement prompt)
converted.append({
    "role": "system",
    "content": REPLACEMENT_SYSTEM_PROMPT  # loaded from config file
})

# First user message (task + terminal state, without the original system prompt)
converted.append({
    "role": "user",
    "content": (task_description + terminal_state).strip()
})
```

### 5.4 Step 5a: Robust JSON Extraction from Assistant Content

This is the most failure-prone step. The assistant content may contain a clean JSON object, or JSON mixed with prose. Implement the following extraction cascade:

```python
import json
import re

def extract_json_from_content(content: str) -> tuple[dict | None, str]:
    """
    Attempts to extract a JSON object from assistant content.
    
    Returns:
        (parsed_json, prose_outside_json)
        parsed_json is None if extraction fails.
        prose_outside_json is any text outside the JSON blob.
    """
    content = content.strip()
    
    # Strategy 1: Direct parse (content is pure JSON)
    try:
        parsed = json.loads(content)
        if isinstance(parsed, dict):
            return parsed, ""
    except json.JSONDecodeError:
        pass
    
    # Strategy 2: Find the outermost {...} using brace matching
    start_idx = content.find('{')
    if start_idx != -1:
        depth = 0
        end_idx = None
        in_string = False
        escape_next = False
        for i in range(start_idx, len(content)):
            c = content[i]
            if escape_next:
                escape_next = False
                continue
            if c == '\\' and in_string:
                escape_next = True
                continue
            if c == '"' and not escape_next:
                in_string = not in_string
                continue
            if in_string:
                continue
            if c == '{':
                depth += 1
            elif c == '}':
                depth -= 1
                if depth == 0:
                    end_idx = i
                    break
        
        if end_idx is not None:
            json_str = content[start_idx:end_idx + 1]
            prose_before = content[:start_idx].strip()
            prose_after = content[end_idx + 1:].strip()
            prose = (prose_before + "\n" + prose_after).strip()
            
            try:
                parsed = json.loads(json_str)
                if isinstance(parsed, dict):
                    return parsed, prose
            except json.JSONDecodeError:
                # Strategy 3: Try fixing common JSON issues
                fixed = fix_common_json_errors(json_str)
                try:
                    parsed = json.loads(fixed)
                    if isinstance(parsed, dict):
                        return parsed, prose
                except json.JSONDecodeError:
                    pass
    
    # All strategies failed
    return None, content


def fix_common_json_errors(json_str: str) -> str:
    """
    Attempt to fix common JSON generation errors from LLMs.
    """
    # Remove trailing commas before } or ]
    json_str = re.sub(r',\s*([}\]])', r'\1', json_str)
    
    # Note: more aggressive fixes (like quote escaping) are risky
    # and may change semantics. Only add fixes here that are
    # demonstrably safe after inspecting failure cases.
    
    return json_str
```

**Tracking:** Log every trace where Strategy 1 fails, and every trace where all strategies fail. After running the full pipeline, report:
- Count and percentage of traces using each strategy
- Count and percentage of total extraction failures
- Sample 50 failures for manual inspection

### 5.5 Step 5b: Building the `thought` Field

Extract `analysis` and `plan` from the parsed JSON and any prose that surrounded the JSON blob:

```python
def build_thought(parsed_json: dict, surrounding_prose: str) -> str:
    """
    Combine analysis, plan, and any surrounding prose into a
    single thought string for the <think> block.
    """
    parts = []
    
    if surrounding_prose:
        parts.append(surrounding_prose)
    
    analysis = parsed_json.get("analysis", "").strip()
    if analysis:
        parts.append(analysis)
    
    plan = parsed_json.get("plan", "").strip()
    if plan:
        parts.append(plan)
    
    return "\n\n".join(parts)
```

The resulting string becomes the `thought` field (or `reasoning_content` field, depending on the training framework — see Section 3.2).

**Do NOT attempt to rewrite the analysis/plan text into different prose.** Preserve the original text verbatim. Rewriting would require an LLM call per turn and introduces noise. The model will learn to reason in natural prose from the content itself, even though the original was written as a JSON value.

### 5.6 Step 5c: Building `tool_calls` from Commands

Because our inference harness uses a persistent shell, the conversion of commands is straightforward. We do not need to simulate shell state, drop special keystrokes, or worry about accumulated working directory. Every command in the Terminus-2 trace is a valid operation in the persistent session and can be preserved as-is.

```python
import hashlib

def build_tool_calls(
    parsed_json: dict, 
    conversation_id: str, 
    turn_index: int
) -> list[dict]:
    """
    Convert the commands array into a single bash tool_call.
    
    Because the inference harness uses a persistent shell, commands
    are preserved faithfully from the Terminus-2 trace. State-modifying
    commands (cd, export, source) will correctly affect subsequent turns.
    
    Args:
        parsed_json: The parsed assistant JSON
        conversation_id: Unique identifier for this trace (e.g., trial_name)
        turn_index: The 0-based index of this assistant message in the conversation
    
    Returns:
        List containing zero or one tool_call dicts.
    """
    commands = parsed_json.get("commands", [])
    
    if not commands:
        return []
    
    # Process keystrokes
    processed_parts = []
    
    for cmd in commands:
        ks = cmd.get("keystrokes", "")
        
        # Skip empty/whitespace-only keystrokes (these are "wait" commands
        # with nonzero duration — there is no tool_call equivalent for
        # "just wait N seconds", so we skip these)
        if not ks.strip():
            continue
        
        # Special keystrokes (C-c, C-d) are preserved as-is.
        # In a persistent shell, these are legitimate operations:
        # C-c interrupts a running process, C-d sends EOF.
        # The harness must interpret these and send the appropriate
        # signal to the tmux/pexpect session.
        if ks.strip() in ("C-c", "C-d"):
            processed_parts.append(ks.strip())
            continue
        
        # Strip trailing newline (the \n is the "Enter" key in Terminus-2;
        # in our tool_call format the command string doesn't include it —
        # the harness adds it when sending to the terminal)
        cleaned = ks.rstrip("\n")
        
        if cleaned:
            processed_parts.append(cleaned)
    
    if not processed_parts:
        return []
    
    # Join multiple commands with newline separator.
    # In a persistent shell, each line is sent sequentially to the
    # session. This preserves the original execution order exactly.
    command_string = "\n".join(processed_parts)
    
    # Generate deterministic tool_call ID
    id_seed = f"{conversation_id}:turn{turn_index}"
    tool_call_id = "call_" + hashlib.sha256(id_seed.encode()).hexdigest()[:24]
    
    return [{
        "function": {
            "name": "bash",
            "arguments": {"command": command_string}  # MUST be a dict, not a string
        },
        "id": tool_call_id,
        "type": "function"
    }]
```

**Design decisions embedded in this function:**

1. **One tool_call per assistant turn**, not one per command. Rationale: the terminal output returns as one blob in the next user message, so a 1:1 mapping between tool_calls and tool results is the only consistent option.

2. **Newline-joined commands.** When multiple commands exist (e.g., `ls -la`, `mkdir -p /output`, `find ...`), they are joined with `\n`. In the persistent shell, the harness sends each line sequentially to the session. This preserves the original Terminus-2 execution semantics exactly.

3. **Special keystrokes (C-c, C-d) are preserved.** Because the inference harness is a persistent shell, these are legitimate operations. `C-c` interrupts a running process; `C-d` sends EOF. The harness implementation must detect these tokens in the command string and translate them to the appropriate tmux `send-keys` or pexpect signal (see Appendix C).

4. **Wait-only commands (empty keystrokes with duration) are dropped.** There is no tool_call equivalent for "wait 5 seconds and observe." The model will learn to issue a command and observe the result; explicit waiting is a harness-level concern.

5. **Deterministic IDs.** Using a hash of `conversation_id:turn_index` ensures that re-running the pipeline produces identical output, which is important for reproducibility and debugging.

### 5.7 Step 5d: Building Tool Results from User Messages

The terminal output from Terminus-2 includes the full terminal screen: command prompts showing the working directory, command echo, output, and the next prompt. Because our harness uses a persistent shell, the prompt lines (e.g., `root@host:/testbed#`) are **accurate contextual information** — they reflect the real working directory of the persistent session. We preserve them faithfully.

```python
def build_tool_result(
    user_content: str, 
    tool_call_id: str
) -> dict:
    """
    Convert a Terminus-2 user message (terminal output) into a
    tool result message.
    
    Terminal output is preserved with high fidelity. The prompt lines
    showing working directory are accurate in a persistent shell context
    and provide valuable contextual signal for the model.
    
    Args:
        user_content: The raw content of the user message
        tool_call_id: The ID of the tool_call this is responding to
    
    Returns:
        A tool result message dict.
    """
    text = user_content
    
    # Remove "Current terminal state:\n" prefix — this is harness framing,
    # not part of the terminal output itself
    prefix_a = "Current terminal state:\n"
    if text.startswith(prefix_a):
        text = text[len(prefix_a):]
    
    # Remove "New Terminal Output:\n" prefix (appears from turn 2 onward)
    prefix_b = "New Terminal Output:\n"
    if text.startswith(prefix_b):
        text = text[len(prefix_b):]
    
    # Strip the "Are you sure?" confirmation prompt if present.
    # This is a Terminus-2 harness artifact, not real terminal output.
    CONFIRMATION_MARKER = "Are you sure you want to mark the task as complete?"
    if CONFIRMATION_MARKER in text:
        text = text[:text.index(CONFIRMATION_MARKER)]
    
    # Strip trailing whitespace/blank lines (the 40-line terminal buffer
    # padding) but preserve internal formatting including prompt lines
    text = text.rstrip()
    
    return {
        "role": "tool",
        "content": [
            {
                "type": "text",
                "text": text
            }
        ],
        "tool_call_ids": [tool_call_id]
    }
```

**Note on content format:** The SWE-agent format uses `content: [{type: "text", text: "..."}]` (a list of content blocks) for tool results. Verify that your training framework's data loader handles this correctly. Some frameworks expect `content` to always be a string. If so, use `"content": text` instead and drop the list wrapper.

### 5.8 Step 5e: Handling `task_complete: true` → Submit

When an assistant message has `task_complete: true`, this signals task completion. The conversion logic is:

1. If the assistant message **also has commands**, emit the bash tool_call and tool result as normal, then emit an additional assistant turn with a `submit` tool_call.

2. If the assistant message **has no commands** (just analysis/plan with task_complete), emit only the submit tool_call.

3. If the trace ends with a **confirmation turn** (the harness asked "Are you sure?" and the model responded with `task_complete: true` again), the confirmation user message is consumed by the tool result of the preceding bash call, and the final assistant message becomes the submit.

```python
def handle_task_complete(
    parsed_json: dict,
    conversation_id: str,
    turn_index: int,
    thought: str
) -> list[dict]:
    """
    Generate the submit tool_call message(s) when task_complete is true.
    
    Returns:
        A list of messages to append (either just the submit assistant message,
        or an empty list if task_complete is false).
    """
    if not parsed_json.get("task_complete", False):
        return []
    
    submit_id_seed = f"{conversation_id}:submit{turn_index}"
    submit_id = "call_" + hashlib.sha256(submit_id_seed.encode()).hexdigest()[:24]
    
    submit_message = {
        "role": "assistant",
        "content": "",
        "tool_calls": [{
            "function": {
                "name": "submit",
                "arguments": {}
            },
            "id": submit_id,
            "type": "function"
        }]
    }
    
    # If there's reasoning specifically about task completion,
    # and this is a commands-free turn, attach the thought here.
    # If this turn also had commands, the thought is already on
    # the preceding assistant message.
    if not parsed_json.get("commands", []):
        submit_message["thought"] = thought
    
    return [submit_message]
```

### 5.9 Full Conversion Loop: Putting It All Together

```python
def convert_trace(row: dict) -> dict:
    """
    Convert a single Terminus-2 trace to SWE-agent format.
    
    Args:
        row: A single row from the HuggingFace dataset, containing
             'conversations', 'trial_name', and other metadata.
    
    Returns:
        {
            "conversations": [...],  # converted messages
            "metadata": {...},       # preserved metadata + conversion stats
            "warnings": [...]        # list of warning strings
        }
    """
    messages = row["conversations"]
    conversation_id = row["trial_name"]
    warnings = []
    converted = []
    
    # --- Step 1: Parse Message 0 ---
    if not messages or messages[0]["role"] != "user":
        return {"conversations": [], "metadata": {}, 
                "warnings": ["Invalid trace: does not start with user message"]}
    
    content0 = messages[0]["content"]
    
    TASK_DELIM = "Task Description:\n"
    STATE_DELIM = "Current terminal state:\n"
    
    if TASK_DELIM in content0:
        remainder = content0[content0.index(TASK_DELIM):]
    else:
        remainder = content0
        warnings.append("TASK_DELIM not found in message 0; using full content as task")
    
    if STATE_DELIM in remainder:
        task_description = remainder[:remainder.index(STATE_DELIM)].strip()
        terminal_state = remainder[remainder.index(STATE_DELIM):].strip()
    else:
        task_description = remainder.strip()
        terminal_state = ""
        warnings.append("STATE_DELIM not found in message 0")
    
    # --- Step 2: Emit system + first user message ---
    converted.append({
        "role": "system",
        "content": REPLACEMENT_SYSTEM_PROMPT
    })
    
    first_user_content = task_description
    if terminal_state:
        first_user_content += "\n\n" + terminal_state
    
    converted.append({
        "role": "user",
        "content": first_user_content
    })
    
    # --- Step 3: Process subsequent (assistant, user) pairs ---
    i = 1  # start after message 0
    turn_index = 0
    
    while i < len(messages):
        msg = messages[i]
        
        if msg["role"] != "assistant":
            warnings.append(f"Expected assistant at index {i}, got {msg['role']}")
            i += 1
            continue
        
        # --- Parse assistant message ---
        parsed_json, surrounding_prose = extract_json_from_content(msg["content"])
        
        if parsed_json is None:
            warnings.append(
                f"JSON extraction failed at index {i}; "
                f"treating entire content as prose with no commands"
            )
            # Emit as assistant message with no tool_calls
            converted.append({
                "role": "assistant",
                "content": msg["content"],
            })
            i += 1
            turn_index += 1
            continue
        
        # Build thought
        thought = build_thought(parsed_json, surrounding_prose)
        
        # Build tool_calls
        tool_calls = build_tool_calls(parsed_json, conversation_id, turn_index)
        
        # Build assistant message
        assistant_msg = {
            "role": "assistant",
            "content": "",  # content goes in thought for thinking model
        }
        if thought:
            assistant_msg["thought"] = thought
        if tool_calls:
            assistant_msg["tool_calls"] = tool_calls
        
        converted.append(assistant_msg)
        
        # --- Process corresponding tool result ---
        if tool_calls and (i + 1) < len(messages) and messages[i + 1]["role"] == "user":
            tool_result = build_tool_result(
                messages[i + 1]["content"],
                tool_calls[0]["id"]
            )
            converted.append(tool_result)
            i += 2  # consumed both assistant and user
        elif tool_calls:
            # Assistant had commands but no following user message
            # (shouldn't happen in valid traces, but handle gracefully)
            warnings.append(f"Assistant at index {i} has commands but no following user message")
            i += 1
        else:
            # No tool_calls (e.g., analysis-only turn with no commands)
            i += 1
        
        # --- Handle task_complete ---
        submit_messages = handle_task_complete(
            parsed_json, conversation_id, turn_index, thought
        )
        converted.extend(submit_messages)
        
        turn_index += 1
    
    # --- Build metadata ---
    metadata = {
        "source_format": "terminus-2",
        "source_dataset": row.get("_source_dataset", ""),  # set by caller
        "source_subset": row.get("_source_subset", None),   # set by caller for Nemotron
        "source_model": row.get("model", ""),
        "task": row.get("task", ""),
        "episode": row.get("episode", ""),
        "run_id": row.get("run_id", ""),
        "trial_name": row.get("trial_name", ""),
        "date": row.get("date", ""),
        "enable_thinking": row.get("enable_thinking", None),  # Nemotron-only; None for OpenThoughts
        "num_turns": turn_index,
        "num_warnings": len(warnings),
        "has_special_keystrokes": any(
            any(
                cmd.get("keystrokes", "").strip() in ("C-c", "C-d")
                for cmd in (
                    extract_json_from_content(m["content"])[0] or {}
                ).get("commands", [])
            )
            for m in messages if m["role"] == "assistant"
        ),
    }
    
    return {
        "conversations": converted,
        "metadata": metadata,
        "warnings": warnings,
    }
```

## 6. Chat Template Compatibility Verification

### 6.1 Why This Matters

The converted data will be tokenized through a chat template (Qwen3.5 family) during training. If the data structure doesn't match what the template expects, training will either crash or produce garbage tokens. Verification is not optional.

### 6.2 What to Verify

After converting a sample batch (e.g., 100 traces), run every converted conversation through the actual chat template and check:

1. **No Jinja exceptions.** The template should render without errors. Common failure: `arguments` is a string instead of a dict, causing `|items` to fail.

2. **Correct role rendering.** Every message should produce the expected `<|im_start|>role\n...<|im_end|>` block.

3. **Tool calls render correctly.** The template should produce `<tool_call>\n<function=bash>\n<parameter=command>\n...\n</parameter>\n</function>\n</tool_call>` blocks.

4. **Tool results render correctly.** The template wraps tool results in `<tool_response>...</tool_response>` inside a user turn.

5. **Thinking blocks render correctly.** The `thought`/`reasoning_content` field should produce `<think>\n...\n</think>` blocks.

6. **Token count is reasonable.** Compare the token count of converted traces to the originals. Expect 1.3x-2.0x expansion due to XML-like tool_call tags and the new system prompt. Flag any trace exceeding the model's max context length.

7. **Special keystrokes render in command parameters.** Verify that `C-c` and `C-d` appear correctly inside `<parameter=command>` blocks and are not mangled by the template.

### 6.3 Verification Script

Write a standalone verification script that:

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen3.5-4B")

for trace in converted_traces:
    messages = trace["conversations"]
    try:
        rendered = tokenizer.apply_chat_template(
            messages,
            tokenize=False,
            add_generation_prompt=False,
            enable_thinking=True,
            tools=TOOL_DEFINITIONS  # the bash + submit tool schemas
        )
        token_count = len(tokenizer.encode(rendered))
        assert "<tool_call>" in rendered or not any(
            m.get("tool_calls") for m in messages
        ), "Tool calls present but not rendered"
    except Exception as e:
        print(f"FAILED: {trace['metadata']['trial_name']}: {e}")
```

**Run this on the full dataset before starting training.** Every failure is a potential training data corruption.

### 6.4 Known Qwen3.5 Chat Template Issues

1. **The `arguments|items` bug.** The initial Qwen3.5 release had a bug where `for args_name, args_value in tool_call.arguments|items` fails when arguments is not a mapping. A fix was committed that adds an `if tool_call.arguments is mapping` guard. Ensure you are using the latest chat template version (commit `ce53c84` or later for Qwen3.5).

2. **The `reasoning_content` vs `thought` field name.** The template checks `message.reasoning_content`. If your data uses `thought`, the template will fall back to looking for `<think>` tags in the content. Either rename the field or embed the thinking content as `<think>...</think>` in the content string itself.

3. **The rolling checkpoint behavior.** The template strips `<think>` blocks from assistant messages that appear *before* the last real user query. This is fine for inference but may affect how training data is tokenized if your training framework applies the chat template. Verify that the thinking content is preserved in the token IDs used for loss computation.

## 7. Quality Filtering

Not all converted traces should be included in the final training set. Apply these filters:

### 7.1 Mandatory Filters (drop the trace)

| Filter | Rationale |
|--------|-----------|
| JSON extraction failed on any assistant message | Corrupted action data; the tool_calls would be wrong |
| Trace has fewer than 2 turns (only system + 1 user) | No agent behavior to learn from |
| Final turn does not have `task_complete: true` | Agent did not complete the task; potentially bad signal |
| Chat template rendering fails | Data is structurally incompatible |

### 7.2 Warning Filters (flag but include, review a sample)

| Filter | Rationale |
|--------|-----------|
| `TASK_DELIM` not found in message 0 | System prompt extraction may be wrong |
| Trace exceeds 32K tokens after conversion | May get truncated during training |
| Any assistant turn had prose outside the JSON blob | The `thought` field may contain content the model shouldn't repeat |
| Any assistant turn had commands but no following user message | Missing tool result |
| Trace contains special keystrokes (C-c, C-d) | Not a problem per se (these are preserved), but worth tracking for analysis of how often they appear |

**Note (change from v1.0):** Traces containing special keystrokes are no longer treated as a quality concern. With a persistent shell harness, `C-c` and `C-d` are valid operations that the model should learn. These traces are flagged only for statistical tracking, not for potential exclusion.

### 7.3 Optional Quality Filters

| Filter | Rationale |
|--------|-----------|
| Drop traces with more than 20 turns | Long traces often contain flailing/backtracking |
| Drop traces with identical repeated commands | Model got stuck in a loop |
| Drop traces where every command is trivial (`echo`, `ls` only) | Low signal-to-noise for learning complex agent behavior |

## 8. Scalability and Performance

### 8.1 Dataset Size

Combined across both sources:

| Source | Traces | Messages (est.) | Raw Size (est.) |
|--------|--------|------------------|-----------------|
| OpenThoughts-Agent-v1-SFT | ~15,200 | ~150K-300K | ~1 GB |
| Nemotron-Terminal-Corpus: dataset_adapters | ~226,000 | ~1M-2M | ~6 GB |
| Nemotron-Terminal-Corpus: skill_based_easy | ~44,800 | ~200K-400K | ~1 GB |
| Nemotron-Terminal-Corpus: skill_based_medium | ~89,300 | ~500K-1M | ~2 GB |
| Nemotron-Terminal-Corpus: skill_based_mixed | ~5,690 | ~30K-60K | ~200 MB |
| **Total** | **~381,000** | **~2M-4M** | **~10 GB** |

Processing 381K traces is a significant workload. Expect 2-6 hours on a single machine with 8 cores, or 30-60 minutes with 32+ cores.

### 8.2 Implementation Recommendations

1. **Process each source/subset independently.** Load each dataset separately and run the conversion pipeline on each. This allows parallelism across subsets and makes it easy to re-run a single subset if issues are found.

   ```python
   # OpenThoughts
   ot_dataset = load_dataset("open-thoughts/OpenThoughts-Agent-v1-SFT", split="train")
   ot_converted = ot_dataset.map(convert_trace, num_proc=8)
   
   # Nemotron subsets
   for subset in ["dataset_adapters", "skill_based_easy", "skill_based_medium", "skill_based_mixed"]:
       nm_dataset = load_dataset("nvidia/Nemotron-Terminal-Corpus", subset, split="train")
       nm_converted = nm_dataset.map(convert_trace, num_proc=8)
   ```

2. **Handle the `dataset_adapters` Parquet size issue.** The `dataset_adapters` subset has very large Parquet row groups (individual groups exceed 300MB). The HuggingFace dataset viewer cannot render this subset. When loading with `datasets.load_dataset()`, this should work fine in memory, but if using streaming mode or custom Parquet readers, be aware of the row group sizes. Consider using `datasets.load_dataset(..., streaming=True)` if memory is a concern.

3. **Use the `convert_trace` function identically for both datasets.** The conversion logic is the same — the only difference is in metadata columns. The function already handles both schemas because it accesses `row["conversations"]` (present in both) and uses `row.get()` with defaults for optional fields.

4. **Handle the `enable_thinking` field.** Present in Nemotron, absent in OpenThoughts. Preserve it in output metadata when present:
   ```python
   metadata = {
       ...
       "enable_thinking": row.get("enable_thinking", None),  # None for OpenThoughts
   }
   ```

5. **Tag the source in output metadata.** Each converted trace should record which dataset it came from, to enable per-source analysis and filtering:
   ```python
   metadata["source_dataset"] = "open-thoughts/OpenThoughts-Agent-v1-SFT"
   # or
   metadata["source_dataset"] = "nvidia/Nemotron-Terminal-Corpus"
   metadata["source_subset"] = "skill_based_mixed"
   ```

6. **Output format.** Save as Parquet (preferred for HuggingFace ecosystem) or JSONL. If using JSONL, ensure one JSON object per line and verify that `arguments` dicts are serialized as nested objects, not escaped strings.

7. **Deduplication of system prompt.** The replacement system prompt is identical across all traces from all sources. Consider whether your training framework supports referencing a shared system prompt rather than embedding it in every example.

### 8.3 Storage Estimates

Assuming average 2KB per message (after conversion, including the system prompt):

- 381,000 traces x 8 messages average x 2KB = ~6 GB uncompressed
- Parquet with compression: ~2-3 GB
- Token count: expect ~5B-15B tokens total across all traces

The JSON extraction cascade (Section 5.4) should be validated independently on each source before full-scale processing, because the teacher models differ:

Report the strategy distribution and failure rate per source. If any source has >10% failure rate, inspect failures and consider adding source-specific fixes to `fix_common_json_errors()`.

## 11. Open Questions for Discussion

These items require decisions that are outside the scope of this conversion specification:

1. **Exact system prompt wording.** The replacement system prompt should ideally match the system prompt used at inference time by the persistent shell harness. Coordinate with the inference pipeline owner. This report provides a placeholder in Section 4.2.

2. **Field naming: `thought` vs `reasoning_content`.** The Qwen3.5 chat template uses `reasoning_content`. Llama-Factory may use a different field name. The training pipeline owner should specify the correct key, and the conversion script should use it.

3. **`content` field format for tool results.** Some training frameworks expect `content` to always be a string, not a list of content blocks. If using `content: "string"` (flat), the conversion is simpler. If using `content: [{type: "text", text: "..."}]` (structured), it matches the SWE-agent format more closely but may need data loader adjustments. Confirm with the training framework.

4. **Mixing with native SWE-agent data.** If you have some native SWE-agent format data, you'll want to ensure the converted data uses the same system prompt, tool schemas, and field names. Alignment should be verified before mixing datasets. Native SWE-agent data from a stateless harness may need different handling — the model should learn both patterns, or native data should be adapted to match the persistent shell assumption.

5. **Whether to include the `submit` tool result.** When the model calls `submit`, should there be a following `role: tool` message acknowledging the submission? If the inference harness produces one, we should include it in training data for consistency. If not, omit it.

6. **Handling the `duration` field.** Currently discarded. If there is future interest in using duration for reward modeling or curriculum learning, it could be preserved in metadata.

7. **Harness-level special keystroke protocol.** The conversion preserves `C-c` and `C-d` as literal strings in the command field. The inference harness must detect and translate these. The exact protocol (e.g., does the harness split on `\n` and check each line for special tokens, or does it use a regex?) should be specified in the harness documentation. See Appendix C for recommendations.

8. **Data mixing ratio across sources.** With ~15K traces from OpenThoughts and ~366K from Nemotron, the Nemotron data dominates 24:1. Decide whether to: (a) use all data as-is (Nemotron's task diversity may justify the imbalance), (b) upsample OpenThoughts, (c) downsample Nemotron subsets, or (d) use a curriculum that phases in different sources. The teacher model difference (GLM-4.6 vs DeepSeek-V3.2) also means the two sources may teach slightly different reasoning styles.

9. **Nemotron `dataset_adapters` task complexity.** The `dataset_adapters` subset transforms Math/Code/SWE problems into terminal format. Some of these may be very long multi-step traces (e.g., building and training ML models). Validate that these convert cleanly and that token lengths don't exceed training context limits. Consider a separate quality review for this subset given its size (226K traces) and domain diversity.

10. **Nemotron license compatibility.** Nemotron-Terminal-Corpus uses CC-BY-4.0. OpenThoughts-Agent-v1-SFT license should also be verified. Ensure the combined dataset's license terms are compatible with your intended use.

### A.3 Worked Example: Cross-Turn State Dependency

This example demonstrates why the persistent shell decision matters. Consider a Terminus-2 trace where turn 1 changes directory and turn 2 relies on it:

## Appendix C: Persistent Shell Harness Implementation Notes

This appendix provides guidance for the inference harness team. The conversion pipeline produces training data that assumes a persistent shell; the harness must fulfill that contract at inference time.

### C.1 Recommended Implementation Approaches

**Option A — tmux (recommended for production):**
This is what Terminus-2 itself uses. Start a tmux session, send commands via `tmux send-keys`, read output via `tmux capture-pane`. Mature, well-tested, handles edge cases like programs that take over the terminal (vim, less, interactive prompts).

```python
import subprocess

class PersistentShell:
    def __init__(self, session_name="agent"):
        subprocess.run(["tmux", "new-session", "-d", "-s", session_name])
        self.session = session_name
    
    def execute(self, command: str) -> str:
        # Send each line as keystrokes
        for line in command.split("\n"):
            if line in ("C-c", "C-d"):
                # Send as control character
                subprocess.run(["tmux", "send-keys", "-t", self.session, line, ""])
            else:
                subprocess.run(["tmux", "send-keys", "-t", self.session, line, "Enter"])
        
        # Wait for output to settle (heuristic or prompt detection)
        import time; time.sleep(1)
        
        # Capture terminal state
        result = subprocess.run(
            ["tmux", "capture-pane", "-t", self.session, "-p", "-S", "-200"],
            capture_output=True, text=True
        )
        return result.stdout
    
    def cleanup(self):
        subprocess.run(["tmux", "kill-session", "-t", self.session])
```

**Option B — pexpect (simpler, pure Python):**
Spawn a long-running bash process, send commands via `.sendline()`, read output via `.expect()`. Simpler to implement, but less robust for programs that take over the terminal.

```python
import pexpect

class PersistentShell:
    def __init__(self):
        self.shell = pexpect.spawn("/bin/bash", encoding="utf-8", timeout=30)
        self.shell.setwinsize(40, 120)  # match terminal dimensions
    
    def execute(self, command: str) -> str:
        for line in command.split("\n"):
            if line == "C-c":
                self.shell.sendcontrol('c')
            elif line == "C-d":
                self.shell.sendcontrol('d')
            else:
                self.shell.sendline(line)
        
        # Wait for shell prompt
        self.shell.expect(r'\$\s*$', timeout=30)
        return self.shell.before + self.shell.after
```

### C.2 Special Keystroke Protocol

The conversion pipeline preserves special keystrokes as literal strings in the command field. The harness must detect and handle them:

| String in command | Meaning | Harness action |
|-------------------|---------|----------------|
| `C-c` | Ctrl+C (interrupt) | Send SIGINT to foreground process |
| `C-d` | Ctrl+D (EOF) | Send EOF to stdin |
| Any other string | Normal command | Send as keystrokes + Enter |

Detection logic: split the command string on `\n`, then check each line. If a line is exactly `C-c` or `C-d` (after stripping whitespace), treat it as a special keystroke. Otherwise, send it as a normal command line.

### C.3 Output Capture Timing

The trickiest part of a persistent shell is knowing when a command has finished. Terminus-2 uses the `duration` field for this — it waits N seconds then captures the screen. The inference harness needs a more robust approach:

1. **Prompt detection:** Watch for the shell prompt (e.g., `root@hostname:/path#`) to reappear after sending a command. This indicates the command has completed and the shell is ready for the next command.
2. **Timeout with retry:** If the prompt doesn't appear within N seconds, capture whatever output is available and return it. The model can decide to wait longer or interrupt.
3. **Avoid hardcoding prompt patterns.** The prompt may change based on the working directory, user, or shell configuration. Use a unique marker (e.g., set `PS1` to include a UUID) at session start for reliable detection.

### C.4 Sandboxing

The persistent shell should run inside a Docker container or similar sandbox. The harness manages the container lifecycle:

1. Start container for each task
2. Start persistent shell session inside container
3. Execute agent turns by sending commands to the session
4. Capture output from the session after each turn
5. On task completion (submit) or timeout, stop and destroy the container

This provides the isolation benefits that mini-swe-agent achieves through stateless subprocess execution, while maintaining the persistent state that the training data relies on.
